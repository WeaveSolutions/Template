# NEXPO Environment Variables Cross-Reference

This document tracks every environment variable in `.env.example` and verifies its usage across the codebase to identify duplicates, missing references, and ensure proper configuration.

**Search Scope**: JavaScript, TypeScript, JSON, Terraform (.tf), and Markdown (.md) files across the entire codebase.

## Status Legend
- **VERIFIED** - Variable found and properly used in codebase
- **DUPLICATE** - Variable appears multiple times (needs consolidation)
- **UNUSED** - Variable defined but not referenced in code
- **MISSING** - Variable referenced in code but not in .env.example
- **NEEDS_REVIEW** - Variable needs further investigation

### PORT
- **Value**: `3000`
- **Status**: **VERIFIED**
- **Usage**: Default port for the Next.js frontend development server. It's the standard port Next.js applications listen on during development (e.g., via `npm run dev`).
- **Files**:
    - `.env.example` (Defines `PORT=3000`)
    - `docs/Ports/Ports.md` (Documents Next.js Frontend on port 3000, development scripts, and Docker Compose examples)
    - `package.json` (Next.js development scripts like `dev` typically use this port by default or can be configured via this variable)
    - `docker-compose.yml` (Often maps container port 3000 to host port 3000 for the Next.js frontend service)
- **Notes**: This `PORT` variable specifically refers to the Next.js frontend. Other backend services use distinct port variables (e.g., `API_PORT` for the main backend, `EXPRESS_PORT`, `MINDSDB_PORT`). The comprehensive port allocation strategy is detailed in `docs/Ports/Ports.md`. The Auth microservice is a separate component and its port (if directly exposed) would not be `3000`.

### NEXT_PUBLIC_API_URL
- **Value**: `http://localhost:8010/api`
- **Status**: **VERIFIED**
- **Usage**: Base URL for the Nexpo backend API, consumed by frontend applications (Next.js, Expo). In local development, this typically points directly to the backend service on port 8010. In production, it points to the public API endpoint (e.g., `https://api.nexpo.com/api`), which is usually routed through Kong API Gateway or a load balancer.
- **Files**:
    - `.env.example` (Defines default development value: `http://localhost:8010/api`)
    - `packages/shared-utils/src/api/client.ts` (Primary consumer for making API calls from shared frontend code)
    - `apps/next/next.config.js` (For build-time configuration and making the variable available to the Next.js application)
    - `apps/expo/.env` or `app.config.js` (Equivalent for Expo applications to consume the variable)
    - `docs/Ports/Ports.md` (Documents development and production values: `http://localhost:8010/api` and `https://api.nexpo.com/api` respectively)
- **Notes**: The `NEXT_PUBLIC_` prefix signifies that this variable is intended for client-side exposure. The note "Used in shared API client with fallback to http://localhost:8000" from previous documentation suggests there might be logic in the API client to attempt connection via Kong Gateway (port 8000) if the direct backend (port 8010) is unavailable or under specific conditions. Production environments always use the main domain, which routes through the gateway.

### ENABLE_AWS
- **Value**: `true` / `false`
- **Status**: **VERIFIED**
- **Usage**: Master switch to enable or disable all AWS cloud provider integrations. When set to `true`, AWS-specific configurations, resources, and services are activated across the platform. This includes Terraform deployments for AWS infrastructure (e.g., via `terraform/providers/aws/main.tf`), AWS SDK usage in backend services, and integrations within tools like MindsDB for accessing AWS data sources.
- **Files**:
    - `.env.example` (Defines default value, e.g., `ENABLE_AWS=true`)
    - `mindsdb-main/.env.example` (Documents its use for MindsDB's AWS integration, e.g., `ENABLE_AWS=false`)
    - `terraform/providers/aws/main.tf` (Typically used as an input variable to control AWS resource provisioning)
    - `terraform/README.md` (Referenced in general Terraform setup and provider configuration)
    - `packages/shared-db/prisma/schema.ts` (Used conditionally, e.g., `if (process.env.ENABLE_AWS?.toLowerCase() === 'true')`)
    - `mindsdb-main/docker-compose.yml` (Passed as an environment variable to MindsDB containers, e.g., `ENABLE_AWS: ${ENABLE_AWS:-false}`)
    - `docs/CI-CD/GitHub-Secrets-Setup.md` (Mentioned in relation to conditionally requiring AWS secrets, e.g., `if ENABLE_AWS=true`)
- **Notes**: This variable acts as a global toggle for AWS. Setting it to `false` is intended to disable all AWS-related functionalities. More granular control over specific AWS services (like RDS, DynamoDB, S3) is often managed by separate `ENABLE_AWS_SERVICE_NAME` flags (e.g., `ENABLE_AWS_RDS`, `ENABLE_AWS_S3`), which typically depend on `ENABLE_AWS` being `true`.
